# 观测与运维 Runbook

## 1. 目标

定义 `sdkworkfs` 的指标、日志、追踪、SLO、告警和常见故障处置流程，确保系统不仅能设计出来，也能长期运行。

更细的 SLI / SLO、告警级别、通知矩阵和规则建议见 `42-SLO与告警规则细化规范.md`。

## 2. 核心观测面

### 2.1 Metrics

- 发布耗时
- Manifest 生成耗时
- 节点预热耗时
- 切换耗时
- 回滚次数
- 缓存命中率
- 对象回源次数
- Git 复制延迟
- 心跳超时数量
- 租约切换次数

### 2.2 Logs

必须按结构化日志记录：

- requestId
- releaseId
- nodeId
- repoId
- env
- action
- result

日志保留建议：

- 在线检索保留 `7-30` 天
- 审计日志归档保留 `180-365` 天
- 关键发布与故障日志支持长期归档

### 2.3 Tracing

关键链路必须支持追踪：

- Git push -> Release 创建
- S3 导入 -> Release 创建
- Release 发布 -> 节点预热 -> 切换
- 写回 -> 冲突检测 -> 仲裁结果

## 3. SLO 建议

| 指标 | 目标 |
| --- | --- |
| Release 创建成功率 | `>= 99.9%` |
| 节点切换成功率 | `>= 99.95%` |
| 已预热切换耗时 | `P95 < 500ms` |
| 节点心跳可用率 | `>= 99.99%` |
| Git 镜像任务成功率 | `>= 99.5%` |

## 4. 告警建议

应至少配置：

- 主节点租约频繁切换
- 节点心跳超时
- 当前版本对象缺失
- 回滚频繁触发
- 缓存磁盘接近上限
- Git 双向同步冲突激增
- S3 回源错误率升高

### 4.1 仪表盘建议

至少提供以下仪表盘：

- 发布总览仪表盘
- 节点健康与心跳仪表盘
- 缓存命中与磁盘容量仪表盘
- Git 复制状态仪表盘
- 切换与回滚历史仪表盘

## 5. 运维 Runbook

### 5.1 主节点失效

步骤：

1. 确认租约丢失和心跳状态
2. 确认镜像节点健康
3. 执行故障转移
4. 验证主写入恢复
5. 记录审计与事件

### 5.2 切换失败

步骤：

1. 检查目标 Release 校验结果
2. 检查预热是否完整
3. 确认健康检查结果
4. 执行回滚
5. 标记失败 Release 并禁止继续扩散

### 5.3 缓存磁盘爆满

步骤：

1. 检查当前和 previous 版本是否被 pin
2. 清理非活跃对象
3. 延迟非关键预热任务
4. 检查异常大对象来源

### 5.4 双向同步冲突

步骤：

1. 暂停相关同步关系
2. 检查主优先级远端
3. 检查是否存在 force push 或回环
4. 人工仲裁后恢复同步

## 6. 正式要求

- 所有告警必须能映射到明确 Runbook
- Runbook 必须支持值班交接
- 关键事件必须可通过指标、日志和追踪三者交叉验证
- 关键运行指标必须支持按租户、应用、环境、节点维度过滤
