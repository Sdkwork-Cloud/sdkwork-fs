# 总体规划

## 1. 重构目标

`sdkworkfs` 的目标是建设一个**全 Rust** 的内容分发与运行平台，统一解决以下问题：

- 多语言项目统一接入：React、Node.js、Java、Python、静态资源、二进制程序
- 多源统一收敛：Git、S3 标准对象存储、本地上传都统一沉淀为 `Release + Manifest`
- 多模式交付：支持挂载访问，也支持同步到本地真实目录
- 多级缓存：支持内存缓存、对象磁盘缓存、物化目录缓存
- 快速切换：支持秒级版本切换和回滚
- 集群能力：支持多节点分发、Git 镜像、对象复制、任务编排
- Git 托管：支持完整 Git server 能力，包括 `clone/fetch/push`、镜像、双向同步、hook 和 LFS

## 2. 范围边界

### 2.1 必做范围

- 全 Rust workspace
- 管理 API 与非管理 API
- Git Smart HTTP 与 Git over SSH
- Git 仓库托管、镜像、双向同步
- Release、Manifest、Object 三层模型
- 节点守护进程 `sdkworkfsd`
- 挂载模式与同步模式
- 多级缓存与本地物化
- 分布式任务调度与审计

### 2.2 延后范围

- 完整 SaaS 计费
- IDE 插件生态
- P2P 对象直传
- 自定义脚本市场
- AI 运行时编排

## 3. 北极星原则

- **全 Rust**
  - 业务二进制、共享库、控制面、数据面、CLI 全部以 Rust 为基线
- **冷热路径分离**
  - 控制面不承担高频文件读写；节点守护进程不承担全局策略真相源
- **发布真相统一**
  - Git、S3、Upload 都不是运行时真相，运行时只消费 `Release + Manifest`
- **本地优先**
  - 节点优先读本地真实目录和缓存，回源是兜底，不是常态
- **原子切换**
  - 任何版本切换必须是目录级原子切换，不允许半切换态暴露给应用
- **可恢复**
  - 中心不可用时，节点仍能运行已物化版本
- **可观测**
  - API、任务、节点、Git、对象存储、缓存都必须可观测

## 4. 核心对象

| 对象 | 说明 |
| --- | --- |
| `Repository` | 代码或内容来源，支持 Git / S3 / Upload |
| `Application` | 逻辑应用实体，定义运行时、交付模式和默认策略 |
| `Manifest` | Release 的文件清单与对象映射 |
| `Object` | 以内容哈希标识的对象，存于对象存储与本地缓存 |
| `Release` | 不可变发布版本 |
| `Activation` | 将 Release 激活到目标节点或目标环境的操作 |
| `Node` | 部署节点或边缘节点 |
| `Operation` | 异步任务记录 |
| `Policy` | 分支保护、审批、缓存、同步、回滚等策略 |

## 5. 总体架构分层

系统拆成五个平面：

1. `Access Plane`
   - `sdkworkfs-gateway`
   - 对外管理 API、非管理 API、Git HTTP 入口、统一认证与路由
2. `Control Plane`
   - `sdkworkfs-control`
   - 资源状态、策略、编排、查询投影、审计
3. `Execution Plane`
   - `sdkworkfs-job-runner`
   - 构建、同步、镜像、激活、回滚、GC
4. `Git Plane`
   - `sdkworkfs-git-server`
   - Smart HTTP、SSH、receive-pack、upload-pack、LFS、hook
5. `Node Plane`
   - `sdkworkfsd`
   - 缓存、物化、挂载、同步、切换、写回

## 6. 默认运行模型

### 6.1 默认仓库模型

- 默认 Git 工作流采用 `trunk-based`
- 默认主干分支为 `main`
- 是否允许 `dev` 作为主干可配置
- 发布推荐使用 `tag` 或受保护分支

### 6.2 默认交付模型

- 生产推荐 `sync` 模式
- 调试、透明路径访问或 POSIX 挂载场景支持 `mount` 模式
- 所有模式都先收敛到本地真实目录

### 6.3 默认缓存模型

- `L1` 内存热点缓存
- `L2` 对象磁盘缓存
- `L3` 物化目录缓存
- `L4` 挂载/同步视图

### 6.4 默认存储模型

- 对象存储协议统一遵循 S3 API 标准
- Git 为版本来源和协作入口
- 运行时读取统一走本地物化目录

## 7. 产品级主链路

### 7.1 Git Push 到发布

```text
git push
-> sdkworkfs-git-server 接收
-> 策略校验 / hook / 审计
-> Repository 更新 resolved revision
-> 触发 build / release operation
-> 产出 Manifest + Object + Release
-> 发布到目标环境
-> 节点预热 / 物化 / 切换
```

### 7.2 S3 / Upload 到发布

```text
upload or import
-> 对象登记
-> 生成 Manifest
-> 创建 Release
-> verify / publish
-> 节点预热 / 物化 / 切换
```

### 7.3 运行时读取

```text
应用读取文件
-> current 指向的真实目录
-> miss 时查 L1 / L2
-> miss 时回源对象存储
-> 回填缓存
-> 写入真实目录
```

## 8. 目录与默认路径

默认路径模型如下：

- 受管 Git 仓库：`./var/sdkworkfs/git/repos`
- Git 镜像区：`./var/sdkworkfs/git/mirrors`
- 构建工作区：`./var/sdkworkfs/build/workspaces`
- 对象缓存：`./var/sdkworkfs/cache/objects`
- 物化目录：`./var/sdkworkfs/materialized`
- 节点元数据：`./var/sdkworkfs/node/metadata`
- 本地运行时数据库：`./var/sdkworkfs/node/state`

这些路径都必须可配置，并支持租户、仓库、应用、节点隔离。
