# 快速切换与回滚机制

## 1. 设计目标

- 版本切换快
- 切换过程对当前版本无侵入
- 切换失败可快速回滚
- 多节点切换可观测、可分批、可恢复

## 2. 切换前提

- 目标 Release 已通过校验
- 目标 Release 所需对象已下载或可按策略懒加载
- 物化目录已准备完毕
- 当前版本仍然保持可用

## 3. 标准切换流程

```text
感知新 Release
-> 拉取 Manifest
-> 下载缺失对象
-> 物化到 releases/<newRelease>
-> 预热热点文件
-> 完整性校验
-> 原子切换 current
-> 标记 previous
```

## 4. 原子切换标准

- Linux / macOS 优先采用符号链接或目录切换
- Windows 优先采用 junction、目录重命名或等价原子更新
- 切换动作必须由专门切换器统一执行

## 5. 回滚标准

回滚触发条件：

- 激活失败
- 健康检查失败
- 关键对象缺失
- 业务人工回滚

回滚动作：

```text
current -> previous
记录回滚原因
保留失败 Release 供排障
```

当前控制面 MVP 已实现的最小人工回滚能力：

- Release 必须先经过 `verify -> publish -> activate` 才会形成稳定回滚点
- 控制面通过 `application.currentReleaseId` 识别当前生效版本
- `POST /api/v1/releases/{releaseId}:rollback` 会把当前版本切回最近一个稳定版本
- 回滚动作会记录操作记录和审计日志，供后续节点切换器与运维界面对接

## 6. 保留策略

- 当前运行版本必须保留
- 前一个稳定版本必须保留
- 最近 N 个版本按策略保留
- 已标记回滚点的版本必须优先保留

## 7. 切换模式

- `immediate`：接收到发布后立即切换
- `scheduled`：按窗口切换
- `manual`：人工确认后切换
- `canary`：按节点比例分批切换

### 7.1 金丝雀切换要求

- 必须支持节点标签选择
- 必须支持比例控制
- 必须支持失败自动停止扩散
- 必须支持按批观察健康指标

## 8. 性能要求

- 已完成预热时，切换应接近原子目录切换时间
- 默认目标：切换操作本身小于 `100ms`
- 回滚操作本身小于 `100ms`

## 9. 失败恢复

- 切换失败不应破坏当前运行版本
- 校验失败必须拒绝切换
- 若部分预热失败，可根据策略选择延迟切换或允许懒加载

## 10. 健康检查与自动回滚

自动回滚可基于以下条件：

- 进程启动失败
- 健康检查连续失败
- 错误率超过阈值
- 启动时关键文件缺失
- 节点本地校验失败

建议配置：

- 回滚观察窗口
- 失败阈值
- 最小成功节点比例
- 手工接管开关
