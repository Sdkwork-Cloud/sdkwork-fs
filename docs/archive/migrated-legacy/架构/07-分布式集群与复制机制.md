# 分布式集群与复制机制

## 1. 目标

- 支持多机器集群
- 支持多环境、多节点分发
- 支持单主默认与多主可选
- 支持节点缓存预热和发布传播

## 2. 节点角色

| 角色 | 职责 |
| --- | --- |
| `primary` | 接收写入、发布、Git push、策略决策 |
| `worker` | 执行下载、缓存、同步、切换 |
| `edge` | 面向业务负载，优先提供读取能力 |
| `mirror` | 用于灾备与跨地域同步 |
| `cache-only` | 只做缓存与对象预热 |

## 3. 默认集群模型

默认：`single-primary`

特点：

- 一个主节点负责版本生成与主写入
- 多节点负责分发和运行
- 更易控制一致性、发布和审计

高级模型：`multi-primary`

特点：

- 多节点可受控接收写入和发布
- 必须显式启用仲裁、冲突策略和同步规则

## 4. 集群数据流

```text
Primary 生成 Release
-> 发布事件广播
-> Worker / Edge 拉取 Manifest
-> 对象预热
-> 本地物化
-> 按策略切换
```

## 5. 一致性模型

- 发布元数据：强一致，依赖中心数据库事务
- 节点缓存：最终一致
- 当前运行版本：节点内强一致
- 多主同步：按策略最终一致或受控仲裁

## 6. 主节点租约与心跳

默认 `single-primary` 模式下必须支持：

- 主节点租约 `leader lease`
- 节点心跳
- 失效检测
- 故障转移冷却时间

推荐规则：

- 主节点通过中心存储维护可续租租约
- Worker / Edge 节点周期性上报心跳
- 控制面必须记录节点最近存活时间
- 租约丢失后，原主节点不得继续接受主写入

### 6.1 租约要求

- 租约必须有过期时间
- 续租必须幂等
- 跨节点提升必须避免双主并发窗口

### 6.2 心跳要求

心跳至少包含：

- nodeId
- role
- currentRelease
- diskUsage
- cacheUsage
- healthStatus
- lastGitReplicationStatus

## 7. 复制标准

- Release 元数据通过中心数据库共享
- 对象通过 S3 共享或镜像复制
- 节点间通过事件总线感知发布与回滚

## 8. 节点同步策略

支持三种节点同步策略：

- `eager`：发布后立即预热对象并物化
- `lazy`：先更新元数据，首次访问再拉取对象
- `warm-on-switch`：接近切换窗口时预热

生产默认建议：

- 核心应用采用 `eager`
- 长尾应用采用 `warm-on-switch`

## 9. 主从切换与灾备

- 主节点失效后，可通过手工或自动故障转移提升镜像节点
- 故障转移不改变已存在 Release 的不可变语义
- 故障转移必须记录审计事件与切换时间点

### 9.1 仲裁要求

多主模式必须显式定义：

- 仲裁来源
- 主优先级
- 冲突拒绝策略
- 最终一致性边界

## 10. 跨地域要求

- 支持主地域与镜像地域
- 支持对象复制和 Git 镜像
- 支持切换到灾备主节点

## 11. 故障恢复

- 主节点不可用时，已激活的节点必须继续提供本地版本访问
- 网络分区时，节点不得破坏当前版本
- 多主模式下，冲突不得默默覆盖

## 12. 可配置项

- `cluster.mode`
- `cluster.primaryRegion`
- `cluster.failoverPolicy`
- `cluster.nodeRole`
- `cluster.releasePropagation`
- `cluster.prefetchPolicy`
- `cluster.leaderLeaseTtl`
- `cluster.heartbeatInterval`
