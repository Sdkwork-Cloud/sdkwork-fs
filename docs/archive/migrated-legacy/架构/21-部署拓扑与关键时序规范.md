# 部署拓扑与关键时序规范

## 1. 目标

把 `sdkworkfs` 的部署方式和关键业务时序固定下来，避免后续实现各模块对“谁先做、谁通知、谁切换”理解不一致。

实际环境准备、依赖安装、启动顺序和验收检查见 `31-安装部署与环境准备规范.md`。

## 2. 拓扑模式

### 2.1 单区域单主拓扑

```text
Client / CI
  -> Git Gateway / Upload API
  -> Control Plane
  -> PostgreSQL
  -> Release Builder
  -> S3
  -> NATS
  -> Worker / Edge / Primary Nodes
```

适用：

- 默认生产模式
- 一个主写节点，多读多分发节点

### 2.2 双区域主备拓扑

```text
Region A: Control Plane + Primary + S3 A + Git Primary
Region B: Standby Control Plane + Mirror Primary + S3 Mirror + Git Mirror
```

适用：

- 跨地域容灾
- 主地域故障时切换

### 2.3 多主拓扑

```text
Primary A <-> Primary B
   |             |
 Git/S3      Git/S3
```

适用：

- 高级协同模式
- 必须启用仲裁和冲突策略

## 3. 关键时序

### 3.1 Git 发布到节点激活

```text
git push
-> Git Gateway 接收
-> Control Plane 校验策略
-> Release Builder 生成 Release
-> 发布事件写入总线
-> 节点接收事件
-> 节点预热和物化
-> 节点执行切换
-> 节点回报结果
```

### 3.2 S3 导入到节点激活

```text
对象导入
-> Release 生成
-> 发布事件
-> 节点拉取 Manifest
-> 对象预热
-> 物化
-> 原子切换
```

### 3.3 自动回滚时序

```text
新版本切换
-> 健康检查失败
-> 触发回滚判断
-> current 指向 previous
-> 标记失败 Release
-> 广播回滚事件
```

### 3.4 主节点故障转移

```text
主节点心跳丢失
-> 租约超时
-> 备节点检查自身健康
-> 提升为新主节点
-> 重新开放主写入口
-> 广播主节点变更事件
```

### 3.5 多主冲突仲裁时序

```text
Primary A 接收写入
-> Primary B 同一路径存在竞争写
-> 仲裁器判断主优先级或冲突策略
-> 拒绝一方或进入人工仲裁
-> 记录冲突审计事件
```

### 3.6 写回冲突处理时序

```text
节点本地写入
-> 写回任务进入队列
-> 远端校验版本
-> 发现冲突
-> 生成冲突副本
-> 更新任务状态为 CONFLICT
-> 等待人工或策略仲裁
```

### 3.7 滚动升级时序

```text
升级控制面
-> 校验节点升级窗口与版本门禁
-> 分批升级 worker
-> 分批升级 edge
-> 最后升级 primary
-> 执行回归验证
```

## 4. 拓扑选择建议

- 小规模生产：单区域单主
- 高可用生产：双区域主备
- 高级联邦：多主

## 5. 正式要求

- 每种拓扑都必须有明确主写入口
- 每条关键时序都必须有幂等边界
- 切换与回滚时序必须可观测和可审计
- 多主拓扑必须有冲突仲裁器或显式拒绝策略
