# 发布策略与环境治理规范

## 1. 目标

统一定义 `sdkworkfs` 的环境通道、发布门禁、审批、冻结窗口、金丝雀、回滚门禁和环境治理规则，确保平台默认行为清晰、可配置、可审计，并且默认面向 trunk-based 开发模式，同时明确支持 release branch、tag 和候选版本流转。

## 2. 环境通道

标准环境：

- `dev`
- `test`
- `staging`
- `prod`

可扩展环境：

- `preview`
- `dr`
- `benchmark`

环境策略必须支持平台默认值、租户覆盖、应用覆盖和环境覆盖四层叠加。

## 3. 发布门禁

正式发布至少要满足：

- Release 校验通过
- 目标环境来源策略允许
- 必需审批通过
- 关键测试通过
- 当前环境无冻结窗口
- 当前无阻塞性同步冲突

### 3.1 来源规则矩阵

| 环境 | 默认来源 | 允许来源 | 默认审批 | 推荐交付模式 |
| --- | --- | --- | --- | --- |
| `dev` | `main` 最新受保护提交 | `main`、受控 feature/ref、临时验证 ref | 无审批或单人审批 | 快速迭代，允许 `mount` 或 `sync` |
| `test` | `main` 或候选提交 | `main`、测试分支、候选 tag | 单人审批 | 稳定验证，建议 `sync` |
| `staging` | 候选 tag | tag、受控 release 分支 | 单人或双人审批 | 接近生产，建议 `sync` 或 `hybrid` |
| `prod` | 正式 tag/release | 仅正式 tag/release | 双人或多角色审批 | 稳定发布，建议 `hybrid` 或受控 `sync` |

说明：

- `dev` 默认在主干开发，这是推荐基线，但必须支持通过配置切换为受保护分支模式
- `prod` 默认不得直接使用未打 tag 的来源
- 来源规则必须能按环境显式配置，而不是从 `dev` 级联默认到 `prod`

### 3.2 环境来源策略建议字段

每个环境至少应支持：

- `sourcePolicy.mode`：`trunk-head` / `protected-branch` / `release-branch` / `release-tag`
- `sourcePolicy.allowedPatterns`
- `approvalPolicy.mode`
- `deploymentPolicy.mode`
- `rollbackPolicy.mode`
- `freezePolicy.calendar`

## 4. 审批策略

支持：

- 无审批
- 单人审批
- 双人审批
- 多角色审批
- 受控自动化审批

生产建议：

- `prod` 至少双人审批或自动化加人工兜底审批

### 4.1 审批角色建议

- 开发负责人
- 运维负责人
- 平台管理员
- 安全审批人
- 值班负责人

### 4.2 推荐审批分工

- `dev`：默认不阻塞主干迭代，但关键配置变更仍可要求单人审批
- `test`：由开发负责人或测试负责人审批
- `staging`：建议开发负责人加运维负责人联合审批
- `prod`：建议开发负责人、运维负责人必选，安全变更场景附加安全审批

## 5. 冻结窗口

必须支持：

- 固定时间窗口冻结
- 节假日冻结
- 事件驱动冻结
- 租户级、应用级、环境级冻结

冻结期间：

- 禁止新的正式激活
- 允许紧急回滚
- 允许受控紧急变更

### 5.1 冻结通知要求

- 冻结开始前应提前通知相关责任人
- 冻结解除后应记录恢复时间与恢复人
- 紧急例外变更必须实时通知值班与平台负责人

## 6. 金丝雀策略

支持：

- 按节点比例
- 按节点标签
- 按地域
- 按租户

金丝雀过程中必须：

- 采集健康指标
- 支持自动停止扩散
- 支持回滚到 stable 版本
- 支持逐阶段人工确认或全自动晋级

### 6.1 金丝雀阶段建议

1. `1%` 或单节点验证
2. `10%` 小批量验证
3. `50%` 半量扩散
4. `100%` 全量发布

### 6.2 停止条件建议

出现以下任一情况时应停止扩散：

- 错误率高于稳定基线阈值
- P95 / P99 延迟明显回退
- 节点切换成功率未达标
- 关键业务探针失败
- 写回冲突量或缓存 miss 突增

## 7. 回滚门禁

回滚默认应当比正式发布更宽松，但仍需要：

- 明确回滚目标
- 记录回滚原因
- 记录审批或自动触发条件
- 保证回滚路径存在可用缓存或可回源对象

### 7.1 紧急回滚

紧急回滚应支持：

- 跳过普通审批
- 保留完整审计记录
- 自动通知值班与负责人
- 自动生成事故编号或关联事件编号

### 7.2 紧急回滚规则

- 必须优先回滚到最近稳定版本，而不是任意历史版本
- 回滚成功后必须进入冻结观察期
- 回滚期间不得并发发起新的正式发布

## 8. 环境隔离

- `dev` 与 `prod` 的缓存、物化目录、写回队列、Git 凭据应逻辑隔离
- `prod` 默认禁止直接使用未打 tag 的不稳定来源
- 跨环境复制、跨环境共享缓存必须显式配置
- 生产环境默认禁止从开发环境继承宽松写权限

## 9. 环境级配额与限制

建议支持：

- 环境级最大并发切换数
- 环境级最大缓存占用
- 环境级最大写回并发
- 环境级最大 Git 同步并发
- 环境级冻结窗口配置
- 环境级对象拉取速率限制

## 10. 正式要求

- 每个环境必须有清晰的来源规则、审批规则和回滚规则
- 环境治理规则必须可配置、可审计、可查询
- 生产环境不得默认继承开发环境的宽松门禁
- 金丝雀、冻结、审批、回滚规则必须能被控制面统一表达

## 11. 补充约束：自动候选发布

- 应用级自动候选发布策略由 `autoReleaseEnabled`、`autoReleaseEnvironment`、`autoReleaseChannel` 表达
- 平台默认值建议为 `defaultEnabled=false`、`defaultEnvironment=dev`、`defaultChannel=CANDIDATE`
- 推荐做法是在 trunk-based 开发下把 `dev` 环境候选发布绑定到主干最新成功同步结果，但是否自动排队必须显式由应用策略或平台默认值控制
- 自动候选发布只负责排队 `BUILD_RELEASE`，不会自动执行 verify、publish、activate，也不会绕过生产环境门禁
