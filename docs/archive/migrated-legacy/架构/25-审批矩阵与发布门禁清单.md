# 审批矩阵与发布门禁清单

## 1. 目标

把审批、门禁、例外、冻结、金丝雀停止条件和回滚条件以清单和矩阵方式固定下来，降低发布过程的人为歧义，并明确哪些步骤必须自动化、哪些步骤必须人工确认。

## 2. 审批矩阵

| 环境 | 默认审批 | 推荐审批角色 | 紧急回滚审批 |
| --- | --- | --- | --- |
| `dev` | 无审批或单人审批 | 开发负责人 | 无审批或值班确认 |
| `test` | 单人审批 | 开发负责人 / 测试负责人 | 无审批或单人审批 |
| `staging` | 单人或双人审批 | 开发负责人 + 运维负责人 | 单人审批 |
| `prod` | 双人或多角色审批 | 开发负责人 + 运维负责人，安全变更附加安全审批 | 单人紧急审批或自动化受控回滚 |

说明：

- 审批矩阵是默认值，必须允许平台级与应用级覆盖
- `prod` 的审批人不得全部来自同一角色组
- 自动化审批只适用于预定义低风险场景，不得替代全部人工审批

## 3. 自动化门禁

下列门禁应优先由系统自动校验：

| 门禁项 | 默认要求 |
| --- | --- |
| Release 完整性校验 | Manifest、对象数量、哈希全部通过 |
| 来源规则校验 | 来源符合环境允许模式 |
| 测试结果校验 | 必需测试任务全部通过 |
| 冻结窗口校验 | 目标环境当前无冻结窗口 |
| 节点可用性校验 | 目标节点成功率、可用率达标 |
| 同步冲突校验 | 当前无阻塞性写回冲突 |
| 审批数量校验 | 审批条数与角色满足规则 |

## 4. 人工门禁

下列场景建议保留人工确认：

- 生产环境首次上线
- 关键配置变更
- 跨地域切换
- 安全热修复
- 金丝雀晋级到 `50%` 或 `100%`

## 5. 发布门禁清单

正式发布前至少检查：

1. Release 已通过完整性校验
2. 来源符合环境规则
3. 审批已通过且未过期
4. 关键测试已通过
5. 当前无冻结窗口
6. 当前无阻塞性同步冲突
7. 当前无主节点故障未恢复事件
8. 当前节点容量、缓存容量和对象回源能力满足要求

## 6. 例外清单

允许例外的场景：

- 紧急回滚
- 重大生产故障修复
- 安全漏洞紧急修补

例外要求：

- 必须记录原因
- 必须记录操作人
- 必须实时通知值班与平台负责人
- 必须在事后补齐审计与复盘

## 7. 冻结窗口例外

冻结窗口内允许：

- 回滚
- 受控热修复
- 明确批准的紧急变更

冻结通知建议：

- 进入冻结前通知相关责任人
- 例外变更实时通知值班与审批人
- 冻结解除后记录解除时间与责任人

## 8. 金丝雀门禁

进入下一阶段前至少检查：

- 健康检查通过
- 错误率未超阈值
- 关键指标未明显回退
- 节点切换成功率达标
- 缓存命中率与对象回源率在预期范围内

### 8.1 停止扩散条件

出现以下任一情况必须停止扩散：

- 错误率超过稳定基线
- P95 / P99 延迟显著升高
- 节点回滚数量异常
- 关键业务探针失败
- Git / 对象存储回源异常

## 9. 紧急回滚清单

紧急回滚前至少确认：

1. 目标回滚版本存在且可用
2. 当前事故编号或原因已记录
3. 关键节点可以访问回滚版本缓存或回源对象
4. 回滚动作已关联负责人和值班人

回滚完成后至少确认：

1. 当前运行版本已恢复稳定
2. 回滚审计已落库
3. 环境进入观察期或冻结期
4. 事故复盘任务已创建

## 10. 正式要求

- 审批矩阵必须和环境治理规则一致
- 门禁清单必须可由系统自动校验一部分
- 人工步骤必须可审计
- 金丝雀停止条件和紧急回滚条件必须被系统显式表达
