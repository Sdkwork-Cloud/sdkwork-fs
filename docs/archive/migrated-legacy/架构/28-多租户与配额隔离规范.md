# 多租户与配额隔离规范

## 1. 目标

统一定义 `sdkworkfs` 的多租户命名空间、隔离边界、共享边界、配额模型和超限治理机制，确保平台可以在同一控制面承载多个租户，而不会在路径、缓存、权限或发布流上相互污染。

## 2. 隔离维度

至少应覆盖以下维度：

- 控制面资源隔离
- 对象存储前缀隔离
- 本地物化目录隔离
- 写回队列隔离
- Git 凭据与同步配置隔离
- 审计与审批数据隔离

## 3. 租户命名空间

推荐每个核心资源都显式带有 `tenantId`。对象与目录建议采用如下前缀：

- 对象存储：`tenants/<tenantId>/...`
- 物化目录：`<root>/tenants/<tenantId>/apps/<appId>/...`
- 缓存目录：`<cacheRoot>/tenants/<tenantId>/...`

## 4. 共享与去重边界

- 逻辑上不同租户的 Release、审批、审计、Git 关系必须隔离
- 内容寻址对象可在存储层去重，但引用关系和权限判断必须按租户隔离
- 不允许跨租户直接共享写回队列、工作目录和凭据

## 5. 配额模型

建议支持以下配额：

- 对象总容量
- 物化目录总容量
- 缓存容量
- Release 数量
- 并发切换数
- 并发写回数
- Git 同步并发数
- API 请求速率

## 6. 配额策略

建议同时支持：

- `soft limit`：超限告警但暂不阻断
- `hard limit`：超限即拒绝新操作
- 临时提升：带有效期的应急扩容

超限后必须明确返回可识别错误码与剩余配额信息。

## 7. 安全要求

- 租户级访问控制必须与 RBAC 绑定
- 租户级凭据建议独立管理，必要时使用独立 KMS 密钥
- 审计日志必须能够按 `tenantId` 检索

## 8. 观测与计费建议

- 关键指标至少按租户维度聚合
- 配额消耗、对象拉取量、缓存命中率、Git 同步量建议按租户统计
- 超限、临时扩容、配额回收必须生成审计事件

## 9. 正式要求

- 所有核心资源必须具备显式租户归属
- 本地目录、对象前缀、凭据和审计必须满足租户隔离
- 配额模型必须支持查询、审计、告警和硬限制执行
