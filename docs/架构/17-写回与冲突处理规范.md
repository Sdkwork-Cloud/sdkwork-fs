# 写回与冲突处理规范

## 1. 目标

定义 `sdkworkfs` 在本地写入、异步写回、冲突检测、冲突保留和人工仲裁方面的标准行为，确保本地可写能力与不可变 Release 不冲突。

## 2. 可写区域

允许写入的区域仅限：

- `data/`
- `workspace/`
- `cache/`

禁止直接写入：

- `releases/<releaseId>/`
- `current/` 指向的不可变版本内容

## 3. 写回模式

### 3.1 LOCAL

- 只写本地
- 不进入远端写回队列

### 3.2 ASYNC_SYNC

- 本地成功即返回
- 后台异步写回远端
- 默认推荐模式

### 3.3 SYNC_STRONG

- 远端确认后才返回成功
- 仅用于明确受控路径和关键配置数据

## 4. 写回状态机

```text
PENDING
-> DISPATCHED
-> UPLOADING
-> VERIFIED
-> DONE
```

失败状态：

- `FAILED_RETRYABLE`
- `FAILED_PERMANENT`
- `CONFLICT`
- `CANCELLED`

## 5. 冲突来源

- 远端版本已变化
- 多节点同时写同一路径
- Git 同步更新覆盖了本地待写路径
- 对象校验与预期版本不匹配

## 6. 冲突检测

推荐检测依据：

- 版本号
- ETag
- 对象哈希
- 最后修改时间
- 节点来源标识

## 7. 冲突处理策略

支持：

- `reject`
- `last-write-wins`
- `prefer-primary`
- `manual`

默认值：`manual`

## 8. 冲突副本规范

默认命名规则：

```text
<name>.conflict-<timestamp>-<nodeId>
```

要求：

- 冲突副本不得覆盖原文件
- 冲突事件必须记录审计
- 冲突副本必须可在管理界面或 CLI 中查询

## 9. 人工仲裁

至少支持：

- 接受本地版本
- 接受远端版本
- 生成合并版本后重新写回
- 放弃本地待写任务

## 10. 幂等与重试

- 写回任务必须具备唯一任务 ID
- 可重试失败不得生成重复写回结果
- 同一路径的并发写回必须受控串行或受控合并

## 11. 正式要求

- 写回系统不得破坏不可变 Release 目录
- 冲突不得静默吞掉
- 人工仲裁结果必须有审计记录和最终状态
