# 缓存策略与淘汰算法规范

## 1. 目标

在 [05-多级缓存与本地物化机制.md](D:/javasource/spring-ai-plus/spring-ai-plus-business/apps/sdkwork-fs/docs/架构/05-多级缓存与本地物化机制.md) 的分层模型基础上，进一步明确缓存准入、预热、淘汰、水位控制、一致性校验和 GC 算法，使缓存系统可以直接指导实现。

## 2. 总体原则

- 缓存首先服务切换、回滚和稳定运行，而不是单纯追求命中率
- 当前版本与上一个稳定版本必须高于普通热点对象的保留优先级
- 缓存命中后仍应保留校验能力，不能把缓存当作绝对真相源
- 多节点默认各自维护本地缓存，不做复杂分布式强一致缓存

## 3. 准入策略

### 3.1 L0 元数据缓存

- 几乎所有 Manifest、路径索引、Release 指针允许进入
- 基于 `TTL + segmented LRU`
- 禁止无限增长

### 3.2 L1 内存热点缓存

建议准入规则：

- 小于 `1MB` 的小文件优先进入
- 启动脚本、配置文件、首页资源优先进入
- 大文件、模型文件、媒体文件默认不进入

推荐算法：

- `TinyLFU` 准入
- `size-aware LRU` 淘汰

### 3.3 L2 对象磁盘缓存

- 所有被访问或预热的对象可进入
- 以内容哈希去重
- 按对象大小计费
- 当前版本、上一稳定版本、预热中版本对象默认标记为 `pinned`

### 3.4 L3 物化版本缓存

- 只保留可直接运行的完整版本目录
- `current`、`previous stable`、`target prefetch` 三类版本必须优先保留
- 其余版本按保留窗口淘汰

## 4. 默认 TTL 与保留建议

| 层级 | 默认 TTL / 保留期 | 说明 |
| --- | --- | --- |
| `L0` 元数据缓存 | `5-30min` | 允许高频刷新 |
| `L1` 热点内存缓存 | `1-10min` | 偏短 TTL，避免热点污染 |
| `L2` 对象磁盘缓存 | 无固定 TTL，以水位与访问时间为主 | 内容寻址对象更适合容量治理 |
| `L3` 物化版本缓存 | 至少保留 `current + previous stable` | 切换与回滚优先 |
| `L4` 写回与工作缓存 | 直到任务完成并超过保留窗口 | 不能提前清理 |

## 5. 预热算法

### 5.1 发布预热

目标：

- 在正式激活前把关键对象拉到 L2/L3

优先级：

1. 启动入口文件
2. 配置文件
3. 首页与索引资源
4. 运行期热点目录
5. 其余对象

### 5.2 运行时自适应预热

建议基于：

- 最近访问频次
- 节点角色
- 应用类型
- 历史切换访问路径

### 5.3 回滚预热

- `previous stable` 必须常驻
- 若 `previous stable` 丢失，则在后台重新补热
- 不允许在高风险发布窗口把回滚版本清理掉

## 6. 淘汰算法

### 6.1 L0 / L1

- 先按 TTL 过期清理
- 再按访问频次与最近访问时间综合淘汰
- 热点配置文件可提高权重，避免被瞬时大流量静态资源挤出

### 6.2 L2

推荐使用双水位淘汰：

- `high watermark`：触发 GC，例如 `85%`
- `low watermark`：回收到该水位，例如 `70%`

淘汰顺序建议：

1. 非 pinned 且长时间未访问对象
2. 非 pinned 的低频大对象
3. 已退休且无引用的版本对象

禁止淘汰：

- 当前版本依赖对象
- `previous stable` 回滚对象
- 正在预热中的目标版本对象
- 正在写回流程中引用的对象

### 6.3 L3

物化版本淘汰顺序建议：

1. 已退休且连续多个发布周期未访问的版本
2. 非稳定版本且未被金丝雀引用的版本
3. 已完成验证、可从 L2 快速重建的临时版本

## 7. 一致性与校验

- L2 / L3 命中后仍应校验 `objectHash`
- Manifest 与对象校验不一致时必须立即失效该对象
- 失效对象必须标记原因，并允许后台重拉
- 节点激活前必须完成关键对象校验

## 8. 脏数据与写回交互

- `release/` 内容不可变，不允许被写路径污染
- `data/`、`workspace/`、`queue/` 只能进入 L4，不得回写到不可变 Release
- 写回冲突对象不得直接覆盖缓存中的稳定对象

## 9. 背压与保护

当出现以下情况时必须触发保护动作：

- 磁盘缓存接近满水位
- 内存热点缓存抖动严重
- 对象存储回源超时率升高
- 写回队列堆积

保护动作建议：

- 降级预热并发
- 限制新大对象进入 L1
- 优先回收非 pinned 对象
- 触发回源限速和对象下载退避

## 10. 可观测指标

至少暴露以下指标：

- 各层缓存容量与使用率
- 命中率与 miss 率
- 淘汰次数
- pinned 对象数量
- 预热耗时
- GC 耗时
- 回源吞吐与失败率

## 10.5 2026-04-06 MVP 增量

- 当前代码已先落地 L2 对象磁盘缓存预取能力，节点可基于 `materialization-plan` 拉取 storage-backed 对象到本地缓存目录。
- 当前代码已补充 `POST /internal/v1/nodes/{nodeId}:materialize`，可在 L2 miss 时回源对象存储，并将 `storageEncoding in [none,gzip,zstd]` 的 release 解码展开到本地 L3 真实目录。
- 当前代码已补充 `POST /internal/v1/nodes/{nodeId}:switch`，可在同一应用下把已物化 release 切换为 `current`，并把旧 `current` 记为 `previous`。
- 当前预取、物化和切换结果都会在节点元数据目录输出 JSON 索引；其中切换状态落在 `<metadata-root>/switches/<nodeId>/<appId>.json`，可供恢复与缓存保护使用。
- 最小闭环要求为“L2 保存对象存储态字节，L3 校验解码后真实大小并写入本地文件，再由 `current/previous` 指针切换运行入口”。
- 节点切换优先使用符号链接别名；Windows 或不支持符号链接的环境回退为别名路径上的 JSON 指针文件。
- 后续阶段继续补齐挂载视图、控制面分布式编排和自动回滚执行器。

## 11. 正式要求

- 缓存必须围绕 `current`、`previous stable` 和目标预热版本设计优先级
- 淘汰策略必须支持水位控制、pin 机制和校验失败重拉
- 任何 GC 不得破坏切换、回滚和写回链路
