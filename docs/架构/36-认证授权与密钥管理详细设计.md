# 认证授权与密钥管理详细设计

## 1. 目标

在安全标准基础上，细化 `sdkworkfs` 的身份模型、认证方式、RBAC、环境隔离、密钥存储和轮换策略，使控制面、Git Gateway、节点和自动化系统都围绕同一套安全模型实现。

## 2. 身份模型

至少支持以下主体：

- 平台管理员
- 租户管理员
- 仓库维护者
- 应用发布者
- 只读审计用户
- CI 服务账号
- 节点服务账号

## 3. 认证方式

### 3.1 人员用户

- 优先 OAuth2 / OIDC
- 必须支持短期访问令牌
- 建议支持 MFA

### 3.2 CI 与自动化

- 使用短期 Service Token
- Token 绑定租户、仓库、环境范围
- 必须支持过期时间和吊销

### 3.3 节点

- 推荐 mTLS
- 次选节点专用签名 token
- 节点注册时必须校验节点身份与节点指纹

### 3.4 Git SSH

- 使用 SSH 公钥
- 必须保存指纹和所属主体
- 必须支持失效与轮换

## 4. RBAC 权限模型

### 4.1 资源域

- 平台
- 租户
- 仓库
- 应用
- 环境
- 节点
- 审计

### 4.2 动作域

- `read`
- `write`
- `publish`
- `rollback`
- `operate`
- `approve`
- `audit`
- `admin`

### 4.3 最小角色矩阵建议

| 角色 | 典型能力 |
| --- | --- |
| `platform-admin` | 平台级读写、租户管理、故障接管 |
| `tenant-admin` | 租户内仓库、应用、环境、凭据管理 |
| `repo-maintainer` | 仓库推送、镜像配置、分支保护管理 |
| `release-operator` | 创建 Release、发布、回滚 |
| `auditor` | 审计查询、只读访问 |
| `node-agent` | 节点注册、心跳、状态上报 |

## 5. 环境级授权

- `prod` 发布权限必须显式配置，不能默认等同于 `dev`
- 审批与发布权限可以分离
- 节点维护权限不等于发布权限

## 6. 密钥与凭据管理

### 6.1 存储原则

- 凭据不得明文写入源码库
- 长期密钥应存储在 KMS / Vault / 密钥服务中
- 节点仅持有运行所需的最小凭据

### 6.2 密钥类型

- S3 Access Key / Secret
- Git SSH 私钥
- JWT / OIDC 配置
- mTLS 证书
- Service Token 签发密钥

### 6.3 轮换要求

- 所有长期密钥必须可轮换
- 轮换必须支持过渡窗口
- 轮换操作必须生成审计记录

## 7. 典型安全流程

### 7.1 用户发布

```text
用户登录
-> 获取短期访问令牌
-> 调用发布 API
-> Gateway 校验身份与权限
-> 进入审批或发布流程
```

### 7.2 节点注册

```text
节点启动
-> 读取节点证书或 token
-> 调用 internal register
-> 控制面校验节点身份
-> 发放节点运行配置
```

## 8. 审计与告警

必须记录：

- 登录与认证失败
- 权限拒绝
- 凭据创建、更新、轮换、吊销
- 节点注册与证书异常
- Git SSH 密钥变更

## 9. 正式要求

- 所有主体都必须有清晰身份与权限边界
- `prod` 权限必须显式收紧
- KMS / Vault / mTLS / Token 轮换能力必须纳入平台标准
- 认证、授权、密钥管理和审计必须统一设计，不得各子模块自定义一套
