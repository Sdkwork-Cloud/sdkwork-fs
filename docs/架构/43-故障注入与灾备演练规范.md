# 故障注入与灾备演练规范

## 1. 目标

定义 `sdkworkfs` 的故障注入、混沌测试、双区域灾备演练和恢复验收标准，确保高可用能力不是停留在架构图上，而是可以定期验证。

## 2. 演练范围

至少覆盖：

- 主节点故障
- 节点心跳超时
- 网络分区
- S3 高延迟 / 不可用
- Git 镜像失败
- 缓存损坏
- 节点磁盘打满
- 错误回滚路径验证

## 3. 演练频率建议

- `P0/P1` 级核心故障：至少每季度一次
- 单节点故障与缓存故障：至少每月一次
- 关键回滚链路：每次大版本前至少一次

## 4. 演练流程

1. 明确演练范围和目标
2. 确认演练环境和保护边界
3. 执行故障注入
4. 观察监控、告警、回滚和恢复行为
5. 记录结果与改进项

## 5. 故障注入样例

### 5.1 主节点故障

```text
停止 primary
-> 观察租约超时
-> 观察 standby 接管
-> 验证主写入口恢复
```

### 5.2 S3 异常

```text
注入对象存储高延迟
-> 观察缓存命中与回源退避
-> 验证 current 版本持续可读
```

### 5.3 缓存损坏

```text
破坏非活跃对象缓存
-> 触发对象校验失败
-> 验证自动重拉与校验恢复
```

## 6. 恢复目标

演练时至少记录：

- 检测时间
- 告警时间
- 切换时间
- 恢复时间
- 业务影响范围

## 7. 验收标准

- 故障必须被监控捕获
- 告警必须按级别通知到位
- 回滚或故障转移必须可完成
- 事后必须形成复盘记录与整改任务

## 8. 正式要求

- 灾备与故障恢复能力必须通过定期演练验证
- 演练过程必须可审计、可回放、可复盘
- 未通过演练验证的高可用能力不得宣称为生产就绪
